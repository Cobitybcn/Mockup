# Artwork Mockups Interaction Patterns

## Purpose

Artwork Mockups is not a collection of pages. It is a collection of worktables.

Each workspace supports an image-led production process. The interface favors direct work with visual material, spatial destinations, and visible results over administrative forms, tables, lists, or command-heavy controls.

## Drag Before Forms

When a task is about moving, ordering, grouping, or assigning visual material, let the user drag the material directly.

Use a form only for information that cannot be expressed through visual interaction. Keep forms subordinate, fold secondary configuration when it is not needed, and make writing areas comfortable when extended input is necessary.

## Images Before Tables

Use images as the primary means of recognition and decision-making. Artworks, mockups, cameras, scenes, videos, and generated content belong in image-led arrangements.

Tables are appropriate only for genuinely tabular information. They must never replace a visual workflow with an administrative one.

## Boards Before Lists

Use boards and visible assignment areas when material must be grouped, ordered, prepared, or sent to a production destination.

Lists may support secondary information, but they must not replace the spatial understanding offered by a board.

## Icons Before Long Text

Use small, familiar icons for repeated local actions, especially on images. Every icon must have an accessible name, predictable placement, and a tooltip when clarification is needed.

Text remains appropriate for unfamiliar decisions and primary commitments. This principle prevents repeated labels from overwhelming imagery; it does not remove necessary language.

## Actions Live on Images

Actions that affect one image belong on that image. Favorite, edit, download, regenerate, remove, and play controls remain contextual and use the shared Glass Action treatment.

Favorites occupy the established top-left position. Secondary image actions occupy the top-right. Essential state must remain available to keyboard and assistive technology.

## Visual Assignment

An assignment area communicates where an artwork, mockup, camera, reference, video, or publication asset belongs. The destination is visible before the gesture, responds during drag and drop, and confirms the result by displaying the assigned material in place.

Use short destination labels. Preserve useful empty space as working capacity.

## Reference Material as Visual Evidence

Reference images are working evidence, not album content and not exact scenes to reproduce. Users upload real material, arrange it spatially, and save the ordered result as a reusable visual intention.

Illustrative examples must remain visibly identified as examples. Only real uploaded references may enter a generation test. Experimental generation connections remain explicitly marked as LAB paths and must not silently alter the established production flow.

## Workspace Mentality

A workspace behaves like a physical studio table. It brings relevant images, decisions, tools, destinations, and results into one coherent production area.

The active material remains visible. Supporting controls stay quiet and nearby. Feedback appears in the current workspace whenever the outcome can be shown in place.

## Horizontal Browsing

Visual collections are browsed horizontally. Carousels preserve Thumbnail Card scale and make comparison possible without converting a collection into a dense table or list.

Movement is user-directed. Navigation controls remain restrained, position is preserved, and the current place is indicated only when useful.

## Vertical Workflow

The overall workflow always progresses vertically. Horizontal galleries exist inside this vertical sequence; they do not turn the application into a horizontal dashboard.

A workspace normally follows this order:

```text
Title
  ↓
Short description
  ↓
Primary Decision Block
  ↓
Carousel
  ↓
Workspace
  ↓
Assignments
  ↓
Results
```

Not every stage must be populated, but the sequence remains the default mental model. New work extends this flow rather than arranging equal-weight modules across a dashboard.

## Feedback and State

Feedback stays close to the action:

- selection appears on the selected item;
- drop readiness appears on the destination;
- progress appears near the initiating image or decision;
- completion updates the image, workspace, counter, or result in place;
- failure explains the problem without discarding the current arrangement.

Routine feedback uses restrained color and concise language. Interruptive confirmation is reserved for consequential or difficult-to-recover decisions.
